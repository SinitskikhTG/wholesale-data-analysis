# Анализ оптовых продаж и ранжирование клиентов (SQL)

## 📌 Описание проекта
**Бизнес-задача:** Построить прозрачный рейтинг оптовых клиентов в разрезе регионов (городов) на основе исторической суммы их покупок.

**Сложность:** Необходимо было учесть в отчете новых клиентов, которые уже заведены в базу данных, но еще не совершили ни одного заказа (сумма покупок = 0), чтобы они корректно отображались в хвосте регионального рейтинга.

## 🛠 Стек технологий
* **База данных:** MariaDB / MySQL
* **Ключевые концепции SQL:**
  * `CTE (WITH)` — архитектурное разделение сложной логики.
  * `LEFT JOIN` — сохранение полной базы клиентов при объединении.
  * `Оконные функции (RANK OVER)` — ранжирование внутри партиций (городов).
  * `Агрегация (GROUP BY)` и `COALESCE` — обработка дубликатов и NULL-значений.

## 📝 SQL-скрипт решения

```sql
WITH client_totals AS (
    SELECT client_id, SUM(amount) AS client_total
    FROM orders
    GROUP BY client_id
)
SELECT 
    clients.name, 
    clients.city, 
    COALESCE(client_totals.client_total, 0) AS total, 
    RANK() OVER (
        PARTITION BY clients.city 
        ORDER BY COALESCE(client_totals.client_total, 0) DESC
    ) AS client_rank 
FROM clients
LEFT JOIN client_totals ON clients.id = client_totals.client_id;
```

# 📊 Транспонирование данных (Условная агрегация в SQL)

## 📌 Описание проекта
**Бизнес-задача:** Подготовка данных для дашборда генерального директора. Требовалось преобразовать строчные данные по выручке в столбцы (Pivot) для сравнения ключевых регионов продаж (Москва и Санкт-Петербург) в одну строку.

**Сложность:** Реляционные базы данных по умолчанию возвращают результаты в виде строк. Для переворота данных без использования тяжелых подзапросов была применена условная агрегация.

## 🛠 Стек технологий
* **База данных:** MySQL / MariaDB
* **Ключевые концепции SQL:**
  * `Conditional Aggregation` — условная агрегация.
  * `JOIN` — объединение таблиц.
  * `CASE WHEN` — логика ветвления внутри запроса.

## 📝 SQL-скрипт решения

```sql
SELECT 
    SUM(CASE WHEN clients.city = 'Москва' THEN orders.amount ELSE 0 END) AS moscow_revenue,
    SUM(CASE WHEN clients.city = 'Санкт-Петербург' THEN orders.amount ELSE 0 END) AS spb_revenue 
FROM clients
JOIN orders ON clients.id = orders.client_id;
```

## 📊 Пример итогового отчета (Вывод скрипта)

| moscow_revenue | spb_revenue |
| :--- | :--- |
| 365000 | 175000 |

---
*Проект демонстрирует навыки оптимизации запросов и подготовки данных для BI-систем.*

# 📁 Восстановление абсолютных путей (Recursive CTE)

## 📌 Описание проекта
**Инженерная задача:** Восстановление полных абсолютных путей к файлам и папкам на основе плоского списка смежности (Adjacency List), где каждый элемент ссылается только на своего непосредственного родителя (`parent_id`).

**Сложность:** Иерархия файловой системы имеет неограниченную глубину вложенности. Стандартными `JOIN` такую задачу решить нельзя, необходимо применение рекурсивных алгоритмов на стороне СУБД.

## 🛠 Стек технологий
* **База данных:** MySQL / MariaDB
* **Ключевые концепции SQL:**
  * `WITH RECURSIVE` — запуск цикла рекурсии.
  * `UNION ALL` — объединение шагов рекурсии.
  * `CONCAT` — склеивание строковых значений.
  * `Self-JOIN` — соединение таблицы с самой собой.

## 📝 SQL-скрипт решения

```sql
WITH RECURSIVE parent AS (
    SELECT 
        id, 
        name, 
        parent_id, 
        name AS full_path    
    FROM linux_files
    WHERE parent_id IS NULL

    UNION ALL

    SELECT 
        lf.id, 
        lf.name, 
        lf.parent_id, 
        CONCAT(p.full_path, '/', lf.name)
    FROM linux_files lf 
    JOIN parent p ON lf.parent_id = p.id
)
SELECT * FROM parent;
```

## 📊 Пример итогового отчета (Вывод скрипта)

| id | name | parent_id | full_path |
| :--- | :--- | :--- | :--- |
| 1 | var | NULL | var |
| 2 | log | 1 | var/log |
| 3 | mariadb | 2 | var/log/mariadb |
| 4 | error.log | 3 | var/log/mariadb/error.log |

---
*Проект демонстрирует понимание работы реляционных связей (Primary Key -> Foreign Key) и умение работать со сложными иерархическими структурами.*

# 🏆 Локомотивы продаж: Поиск Top-1 товара (Оконные функции)

## 📌 Описание проекта
**Бизнес-задача:** Для оптимизации маркетингового бюджета необходимо выявить товары-лидеры (принесшие наибольшую выручку) строго по одной позиции в каждой товарной категории. 

**Сложность:** Требуется объединить агрегацию данных с ранжированием внутри независимых партиций, избежав дублирования или потери данных из связанных таблиц.

## 🛠 Стек технологий
* **База данных:** MySQL / MariaDB
* **Ключевые концепции SQL:**
  * `Window Functions (RANK OVER)` — ранжирование внутри партиций.
  * `CTE (WITH)` — архитектурное разделение сложной логики.
  * `Multiple JOINs` — объединение нескольких нормализованных таблиц.
  * `GROUP BY` — группировка данных.

## 📝 SQL-скрипт решения

```sql
WITH ranked_products AS (
    SELECT 
        c.name AS `Название категории`, 
        p.name AS `Название товара`, 
        SUM(o.amount) AS `Общая сумма выручки с этого товара`, 
        RANK() OVER (
            PARTITION BY c.name
            ORDER BY SUM(o.amount) DESC
        ) AS rank_products
    FROM categories c 
    JOIN products p ON c.id = p.category_id 
    JOIN orders o ON p.id = o.product_id 
    GROUP BY c.name, p.name
)
SELECT 
    rp.`Название категории`, 
    rp.`Название товара`, 
    rp.`Общая сумма выручки с этого товара` 
FROM ranked_products rp 
WHERE rp.rank_products = 1
ORDER BY rp.`Общая сумма выручки с этого товара` DESC;
```

| Название категории | Название товара | Общая сумма выручки с этого товара |
| :--- | :--- | :--- |
| Электроника | Ноутбук | 160000 |
| Одежда | Куртка | 15000 |
| Книги | Властелин Колец | 4000 |

---
*Проект выполнен как комплексное задание по сборке витрины данных из нормализованных таблиц.*
